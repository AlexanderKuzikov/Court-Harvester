# Court-Harvester — CONTEXT

> **Архитектурные решения, обоснования и соглашения проекта.**
> Последнее обновление: Июль 2026

---

## 🎯 Назначение проекта

**Court-Harvester** — CLI-инструмент для создания и поддержания эталонной базы судов Российской Федерации. База используется как справочник для юридических/документальных систем (в связке с проектами типа `Court-Viewer`, `DocuMind`, `DOCX-Builder`).

Ключевое отличие от одноразовых парсеров: **инкрементальное обновление** — база живёт и evolves вместе с судебной системой РФ (новые участки, переименования, закрытия, новые регионы).

---

## 📐 Принципы архитектуры

### 1. API-first подход

Единственный источник данных — **DaData API (suggestions.dadata.ru)**. Не HTML-парсинг, не GAS (ГАС «Правосудие»), не PDF. Причины:

- **Стабильность**: REST API с контрактом, а не вёрстка, которая меняется
- **Легальность**: DaData — официальный агрегатор данных из ФНС, Росстата и других гос. источников
- **Простота**: suggestions/court — один эндпоинт, POST, JSON → JSON
- **Ограничение**: 10 000 запросов/день. Отсюда — многовекторная стратегия сбора (фазы, ключи, rate limiting)

### 2. Стратегия «умного перебора»

DaData API не позволяет получить все суды «одним запросом». Решение — **многоуровневый поиск с расширением**:

```
Уровень 1: Одна буква (33 запроса, А–Я)
    ↓  если результатов = 20 (MAX), префикс «горячий»
Уровень 2: Две буквы (33×33 = 1 089 запросов)
    ↓  если снова MAX — ещё более горячий
Уровень 3: Три буквы (только горячие префиксы)
```

**Альтернативы, которые рассматривались:**
- **Поиск по городам** — даёт, но не全覆盖 (некоторые районы не привязаны к крупным городам)
- **Прямой перебор RRTT0000–RRTT9999** — 10 000 комбинаций на префикс × 396 префиксов = невозможно
- **Поиск по адресам/улицам** — данные не всегда актуальны

Выбранный компромисс: алфавитный перебор дал 7 761 суд за 3 фазы, а анализ пропусков (фазы 5, 7) — ещё 2 317.

### 3. Множественные API-ключи

Дневной лимит DaData — 10 000 запросов. Полное обновление требует ~18 890. Решение — **KeyRotationManager**:

- Каждый ключ = 9 500 запросов (500 — буфер на ошибки/ретрансляцию)
- До 4 ключей = ёмкость ~38 000 запросов
- 1.env — резервный, не участвует в обновлениях (может использоваться для CLI `harvest` команд)
- Файлы .env в `keys/` — минималистичный формат, парсинг без dotenv

### 4. Rate limiting с резервуаром

DaData не документирует точные лимиты, но на практике:
- 20 запросов/секунду — безопасно
- 50ms между запросами — консервативно

Bottleneck настроен с **token bucket** (reservoir): 20 токенов, пополнение каждую секунду. Это даёт равномерную нагрузку без всплесков.

### 5. Чекпоинты и устойчивость к падениям

Долгие обновления (30–60 минут) должны переживать:
- Разрыв соединения
- Исчерпание ключей
- Нештатное завершение

**Механизмы:**
- Автосохранение каждые 100–200 запросов в `courts_update_progress.json`
- Лог с таймстемпами — можно восстановить последовательность
- `_status` поле (`existing`/`new`/`updated`/`not_found`) → прогресс не теряется

## 🗂 Организация кода

```
src/
├── index.ts                  CLI точка входа (commander)
├── types/
│   └── dadata.ts             Типы DaData API + все 14 типов судов
└── core/
    ├── ApiClient.ts          HTTP-клиент + rate limiter + retry + ошибки
    ├── KeyRotationManager.ts Ротация ключей
    ├── RegionHarvester.ts    Сбор по одному региону (городские запросы)
    └── FullHarvester.ts      Полный сбор (алфавитный перебор)
```

**Важно:** папка `src/examples/` содержит не просто примеры, а **операционные скрипты**:
- `update.ts` — основной скрипт обновления (mode: search | full)
- `test-full-phase*.ts` — исторические фазы сбора
- `analyze-prefixes.ts`, `find-gaps.ts`, — аналитические утилиты
- `test-single-court.ts` — отладка/проверка одного суда

Это архитектурный долг — скрипты выросли как временные, но остались основными.

## ⚖️ Известные компромиссы

### 1. API-ключи в .env файлах (не в переменных окружения)

```bash
# keys/2.env
DADATA_API_KEY=xxx
DADATA_SECRET_KEY=yyy
```

**Почему не переменные среды:** нужно управлять несколькими ключами, а не одним. `.env` в папке `keys/` — простейшее решение с изоляцией.

**Риск:** ключи на диске — поверхность атаки. Митигация: `.gitignore` → ключи не попадают в репозиторий.

### 2. TypeScript без компиляции (tsx)

**Почему:** проект небольшой, `tsx` быстрее, чем `tsc` → `node`. Для CLI-утилиты это приемлемо. `tsc` настроен, но используется скорее для type-checking.

### 3. Дублирование логики в фазовых скриптах

Каждая фаза (phase4–phase9b) — это фактически копия одного паттерна:

```
загрузить → итерировать → проверить API → сохранить
```

**Цена:** сложно поддерживать, легко ошибиться. **Выгода:** фазы запускались по одной с разными параметрами, и изоляция позволяла не бояться сломать предыдущий результат.

### 4. Нет тестов

Vitest в зависимостях есть, тестов — нет. Проект создавался как исследовательско-утилитарный, где «тест» = запуск на реальном API. Для продуктивного использования тесты нужны (см. Советы).

### 5. Терминология «фаз»

Фазы (phase 1–9B) отражают историю сбора и не имеют жёсткой привязки к коду. phase9b ≠ 9-я фаза B, а «Фаза 9 для топ-20 регионов».
